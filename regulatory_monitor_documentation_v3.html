System Overview

The Regulatory Change Monitor Agent is an AI-powered system that automatically tracks, classifies, and analyzes US life insurance regulatory changes for a specific insurance carrier. It monitors federal and state-level sources, determines which regulations are relevant to the carrier's products, maps impacts to specific policy forms, and generates compliance memos with state-specific action items.

How It Works — End to End

Step 1 — Scraping: The Crawl Agent orchestrates 4 scrapers to fetch regulatory data from Federal Register API, Texas Department of Insurance (TDI), California Department of Insurance (CDI), and OFAC SDN list. All scraped data is stored in a SQLite database.

Step 2 — Deduplication: The Compare Agent checks each scraped item against existing database records using unique source IDs. Duplicates are skipped, only new items are stored.

Step 3 — Classification: The Filter Agent sends each regulation to Claude AI, which classifies whether it's relevant to life insurance, assigns a severity (critical/high/medium/low), and determines which state it applies to (TX/CA/BOTH).

Step 4 — Impact Mapping: The Impact Agent takes the relevant regulations and searches ChromaDB (which contains all carrier policy documents indexed as vectors) to find which specific policies are affected.

Step 5 — Memo Generation: The Draft Agent generates a full compliance memo for each regulation that has policy impacts, including separate TX and CA action items, deadlines, and recommended next steps.

Step 6 — Dashboard: A Streamlit dashboard with 5 pages visualizes all results. The dashboard reads from the pre-computed SQLite database. No AI calls are made at dashboard time.

Policy Files — How They Are Created & Structured

Demo Note: For this POC demonstration, the 18 policy documents were generated with the help of an LLM (Claude AI) based on publicly available information about life insurance product types, standard regulatory requirements, and TX/CA compliance obligations. The product types, regulatory requirements, and compliance sections are accurate to the industry — but the specific product names and form numbers are representative, not sourced from the carrier's actual filed documents.

How Policy Files Would Be Sourced in a Real Engagement

In a real client engagement, the policy documents would come from two authoritative sources:

  • Source 1: SERFF Filing Access (Public) — The NAIC's System for Electronic Rate and Form Filing provides free public access to all filed policy forms. You search by company name or NAIC code, select "Life" business type, and download the filed form PDFs. For this carrier, a SERFF search returned 51 filings in Texas alone — each containing the actual approved policy form PDFs with real form numbers, exact legal language, and state-specific variations.
  • Source 2: Carrier's Internal Document Repository — The carrier's compliance/legal team provides their current policy forms directly from their document management system (SharePoint, internal DMS, or policy administration system). These are the master copies with the most current endorsements and amendments.

Once real policy PDFs are obtained from either source, they would be processed using AI Document Intelligence (OCR + extraction) to convert them into the structured text format the system requires. The rest of the pipeline (ChromaDB indexing, agents, dashboard) works identically — only the input policy files change.

Policy File Structure

Each policy file is a structured text document with three sections:

  • GENERAL section: Product description, face amounts, available terms, features, conversion privileges, beneficiary provisions — information common to all states
  • TEXAS (TX) section: TX-specific compliance requirements — TDI filing details, TX free look periods, TX nonforfeiture standards, TX State Securities Board requirements (for variable products)
  • CALIFORNIA (CA) section: CA-specific compliance requirements — CDI filing details, Proposition 103 rate review, domestic partner provisions, Cal-COBRA requirements, SB 263 annuity suitability rules

This three-section structure is critical — it allows the Impact Agent to determine not just which policy is affected, but which state-specific section of that policy needs attention.

ChromaDB Vector Indexing

ChromaDB is a vector database that converts text into numerical representations (embeddings) and enables similarity-based search. It's used to find which policy documents are related to a given regulation.

How Policy Indexing Works

Embedding model: all-MiniLM-L6-v2 — a lightweight sentence transformer that runs locally on CPU. No API cost, no external calls. It converts text into a 384-dimensional vector.

Chunking strategy: Each of the 18 policy files is split into exactly 3 chunks based on the section markers in the file:

  • GENERAL chunk — product description, face amounts, terms, features
  • TX chunk — Texas-specific compliance requirements
  • CA chunk — California-specific compliance requirements

Total indexed: 18 policies × 3 chunks = 54 chunks stored in ChromaDB

Metadata per chunk: Each chunk carries metadata — policy_id (e.g., P01), policy_name, form_number, and section (GENERAL/TX/CA). This metadata is returned alongside search results so the agent knows exactly which policy and which section matched.

Persistence: The ChromaDB index is saved to disk. Once indexed, it doesn't need to be re-created.

How the Impact Agent Uses ChromaDB

When the Impact Agent processes a relevant regulation (e.g., "New OFAC sanctions requirements for financial institutions"), it:

  1. Takes the regulation title + text and sends it to ChromaDB as a search query
  2. ChromaDB converts the query into a 384-dimensional vector using the same embedding model
  3. ChromaDB performs cosine similarity search across all 54 policy chunks and returns the top 10 most similar chunks
  4. The returned chunks might include: P09 Universal Life (OFAC sanctions clause), P05 IUL (compliance section), R06 Overloan Protection (general provisions), etc.
  5. The agent sends both the regulation AND the matched policy chunks to Claude AI
  6. Claude confirms which policies are truly affected and identifies the specific clauses — eliminating false positives from the vector search

This two-stage approach (vector search → AI confirmation) combines the speed of vector search with the accuracy of AI judgment.

Severity Classification — How the AI Decides

The Filter Agent sends each regulation to Claude AI along with a structured prompt that defines exactly how severity should be assigned. The prompt contains a classification rubric that Claude follows when analyzing each regulation:

Severity Classification Rubric (from the AI prompt)
SeverityCriteria (what triggers this level)Examples
CRITICAL Direct regulatory mandate requiring immediate policy form changes, filing deadlines, or sanctions compliance. Has a specific effective date. Non-compliance carries penalties or enforcement action.
  • OFAC SDN list updates (sanctions compliance — all policies affected immediately)
  • New state law with specific effective date requiring form re-filing
  • Emergency TDI/CDI bulletins with compliance deadlines
  • Final rules on nonforfeiture standard changes
HIGH Significant regulatory change that will likely require policy amendments, new filings, or process changes within 6–12 months. Not yet effective but impact is certain.
  • New suitability rules for annuity/life products (SB 263)
  • Changes to nonforfeiture calculation standards
  • New disclosure requirements for illustrations
  • Legislative changes enacted but not yet effective (SB 622, SB 1238)
MEDIUM Regulatory update that may require monitoring or minor adjustments but no immediate action. Proposed rules still open for comment, or guidance that clarifies existing requirements.
  • Proposed rules open for public comment (not yet final)
  • Guidance bulletins clarifying existing regulations
  • NAIC model law updates under consideration
  • Industry studies or data collection requests
LOW Informational or tangentially related — good to know for awareness but no compliance action needed. Does not directly affect the carrier's policy forms or operations.
  • Enforcement actions against other carriers (industry awareness)
  • Research studies on insurance markets
  • Agency information collection notices
  • Other carriers' form filing approvals or denials

How the prompt works: The Filter Agent sends this rubric as part of the system prompt to Claude AI for every regulation. Claude reads the regulation's title and abstract, applies the rubric, and returns a JSON response with: relevant (true/false), severity (critical/high/medium/low), state (TX/CA/BOTH), and reason (explanation of why it classified this way). The classification is AI judgment informed by the rubric — not a keyword-matching rules engine.

5-Agent Pipeline & Data Flow

The system uses 5 custom Python agents. No external agentic framework (like CrewAI or LangChain agents) is used — all agents are hand-written for full control over prompts, error handling, and cost tracking. Three agents use Claude AI via the Anthropic API, two are pure Python logic. Below, each agent is explained with the data it saves.

Agent 1: Crawl Agent No AI

Purpose: Orchestrates all 4 data scrapers and collects regulatory data from external sources.

Scrapers it runs:

  • Federal Register API — REST API, 12 search terms, paginated. Fetches title, abstract, date, agency, URL. → 3,367 documents
  • TDI Bulletins (Texas) — HTML scraping from TDI website, years 2021–2026. → 91 bulletins
  • CDI Bulletins (California) — HTML scraping from CDI website, bulletins + notices. → 74 items
  • OFAC SDN List — CSV download from US Treasury. 18,713 entries parsed into 1 snapshot record. → 1 record

Agent 2: Compare Agent No AI

Purpose: Prevents duplicate records when scrapers run multiple times. Each regulation has a unique source ID (Federal Register document number, TDI bulletin number, etc.). Before inserting a new record, the Compare Agent checks if that source ID already exists. Only genuinely new items are inserted.

Why this matters: In a real scenario, scrapers run daily. Without deduplication, the same regulation would be inserted every day.

Agent 3: Filter Agent Uses Claude AI

AI Model: Claude Sonnet 4 via Anthropic API

Purpose: Classifies each regulation as relevant or not relevant to life insurance. For relevant items, assigns severity and determines state applicability using the rubric described in Section 4.

Process: Takes each regulation's title + abstract → sends to Claude with the classification prompt → Claude returns JSON with relevant, severity, state, reason → agent updates the database.

Agent 5: Draft Agent Uses Claude AI

AI Model: Claude Sonnet 4 via Anthropic API

Purpose: Generates a full compliance memo for each regulation that has at least one policy impact.

Process: Fetches regulation + all its policy impacts from database → sends to Claude with memo generation prompt → Claude generates structured memo with 6 sections → save memo text to memos table.

The 6 memo sections:

  1. Regulation Summary — plain English explanation of what changed
  2. Affected Policies — list of specific policies with form numbers and states
  3. Texas Action Items — what the TX compliance team needs to do
  4. California Action Items — what the CA compliance team needs to do
  5. Deadline — when compliance is required
  6. Recommended Next Steps — coordinate with Legal, file amendments, etc.

Skipped items: If a regulation was classified as relevant but the Impact Agent found no specific policy impacts, no memo is generated.

Dashboard — 5 Pages

The dashboard is built with Streamlit (Python web framework) + Plotly (interactive charts). All data is read directly from the SQLite database. No Claude API calls happen when viewing the dashboard — it's a read-only visualization layer over pre-computed results.

Page 1: Home Executive Overview

What it shows: The first page a viewer sees. Provides a high-level summary of the entire analysis.

  • KPI cards row: Large metric cards showing total tracked, relevant to life insurance, policy impacts, memos generated, and policies affected
  • Severity donut chart: Breakdown of relevant regulations by severity — critical (red), high (orange), medium (blue), low (gray). Gives an instant sense of urgency.
  • State comparison bar chart: How many regulations affect TX only, CA only, or BOTH states
  • Source distribution bar chart: How many relevant regulations came from each data source
  • Recent critical & high items table: Most recent urgent regulations with date, title, state, severity, and source

Sidebar filters: State (All/TX/CA), Severity (multi-select), Source (multi-select). Changing any filter updates all charts and tables instantly.

Page 2: Timeline Regulatory Change Over Time

What it shows: A 5-year interactive bar chart showing when regulatory changes occurred.

  • Quarterly stacked bar chart: Quarters on X axis, regulation count on Y axis. Bars stacked and color-coded by severity, source, or state depending on toggle
  • "Color by" toggle: Switch between Severity view, Source view, or State view without changing the data — just the coloring perspective
  • Summary metrics: Total in view, critical count, high count, average per quarter
  • Yearly breakdown table: One row per year with severity counts

Performance: Data is cached for 10 minutes — subsequent page loads are instant.

Page 3: Impact Matrix Policy-Level Analysis

What it shows: Which of the 18 policies face the most regulatory compliance pressure. The most valuable page for compliance teams.

  • Most affected policies bar chart: Horizontal bars ranking all policies by number of impacts. Longest bar = highest priority for compliance work
  • TX vs CA comparison: Two side-by-side charts showing impact counts per policy per state. Shows which state drives more work for each product
  • Impact heatmap: Color grid with policies × quarters. Darker = more impacts. Reveals temporal patterns
  • Policy detail expanders: Click any policy to see every regulation affecting it with state, clause, and impact description
Page 4: Compliance Memos AI-Generated Action Items

What it shows: Full text of every AI-generated compliance memo. The proof of AI value.

  • Memo selector dropdown: All memos listed with severity/state badges and date. Select any to view
  • Memo header: Severity badge, state badge, source, date
  • Full memo display: Complete structured memo — Regulation Summary, Affected Policies, TX Action Items, CA Action Items, Deadline, Next Steps
  • Affected policies sidebar: Policy cards on the right showing which products are affected
  • Download button: Save memo as text file for sharing with compliance teams

Sidebar filters: State (All/TX/CA/BOTH) and Severity (multi-select) to narrow down the memo list.

Page 5: Data Explorer Full Transparency

What it shows: Complete access to all underlying data and processing statistics.

  • Processing funnel: Visual funnel showing data flow — total scraped → relevant → policy impacts → memos. Shows the filtering ratio at each stage
  • Data by source: Metric cards showing relevant regulation counts per source
  • API usage statistics: Per-agent breakdown of API calls, tokens, cost, and time. Full transparency on AI resource consumption
  • Searchable regulation table: Full table of all regulations with text search. Status column shows whether each item was relevant or rejected — making AI decisions transparent and auditable
